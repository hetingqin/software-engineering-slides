敏捷开发的本质

Agile ≠ 工具或方法论，而是一套指导决策的价值观

敏捷开发：价值观与原则

很多人对敏捷开发存在误解，认为：

  • 敏捷就是 Scrum、极限编程 (XP)
  • 敏捷就是每天开站立会议、两周一个 Sprint
  • 敏捷就是用看板软件（如 Jira），或者把便签贴白板上
实际上，敏捷不是一种工具或框架，也不是一套具体的过程，而是一套价值观和原则。各种框架和工具是“术”（手段），敏捷本身是“道”。

敏捷软件开发宣言（2001）

敏捷宣言四大价值观

🤝
个体和互动
高于流程和工具
人是最重要的因素
💻
工作的软件
高于详尽的文档
可运行的成果最有说服力
🤲
客户合作
高于合同谈判
持续沟通比签合同更重要
🔄
响应变化
高于遵循计划
拥抱变化而非抵制变化
宣言右边的内容（流程、文档、合同、计划）正是瀑布模型的核心。敏捷并未否定瀑布右侧的价值，但更看重左侧。

用敏捷的方式“盖房子”

解决跨度长周期、发布烦、变更难的问题
  • 需求："作为一个上班族，我想要一个卧室来休息"（用户故事替代详尽文档）
  • Sprint 1：搭了个草棚（最简可用，客户立刻入住体验）
  • Sprint 2：升级成简易厨房，修复屋顶漏水 Bug
  • Sprint 3：升级成小木屋（由于自动化测试不完备，可能忘了加窗户）
  • Sprint 4：修好窗户，完成砖瓦房。此时客户发现空间不够，决定扩建为 3 居室（拥抱变更）
在不断的短周期迭代和修补中交付。不再等到测试阶段才发布，客户随时能看到可用版本。

瀑布模型与敏捷开发的具体差异

实施环节瀑布模型敏捷开发
需求分析产出详尽需求文档，需求冻结“用户故事”（一句话卡片），边开发边当面沟通细节
架构设计开发前完成完整系统设计渐进式架构，只做适合当前需求的架构，需定期重构
质量保证专门测试阶段集中测试保障依赖自动化测试（单元/集成），前期可能不稳定(如Win10)
发布部署集中测试通过后手工部署生产持续集成与持续发布（CI/CD），代码一交就自动构建部署
迭代节奏如“小瀑布”，前期轻松后期紧张一个个循环的Sprint节奏恒定，做不完的故事不影响发布

我们该不该选择敏捷开发？

1. 团队要小
人数超过规模就要分拆
2. 紧密协作
团队内部协作透明
客户深度配合
3. 扁平化结构
领导支持，更少干预控制
给项目成员主动权
4. 工程基础
写代码必须有自动化测试
搭建好源码管理和CI环境
软件开发最核心的是人，而不是用什么方法。如果不具备全面敏捷的条件，可以先借鉴持续集成、每日站会等好实践。

分而治之的大厂策略

大项目拆成小项目，大服务拆成微服务，大团队拆成小团队。
技术底座
底层依赖**微服务、容器**等新技术提供隔离，避免互相牵制
通信协作
团队之间依赖**标准协议通信**配合，边界清晰
组织结构
组织**扁平化**，工程师地位够高，大企业也能真正敏捷起舞

一切工作围绕 Ticket（看板）开展

  • 所有的工作——不管是报 Bug、提需求，还是重构代码——都必须发起一个 Ticket
  • 看板分为 To Do、In Progress、Done 栏
  • 每天上班第一件事，就是根据看板直观掌握 Sprint 的进度
  • 员工自己找任务认领，项目经理看积压随时发现风险

现代数字看板（Jira）

取代了糊在墙上的便签和甘特图

基于 Git 和 CI 的自动化交付与部署

解决“代码不稳定”和“部署太麻烦”两大痛点

代码准入与审查（Code Review）

  • 不再直接提交到主干（master）
  • 开分支（branch）开发任务
  • 提交合并请求（Pull Request / PR）
  • 要求同行检视代码变更并给出评价

持续集成（CI）工具机器人

  • PR 提交后，CI 自动下载、依赖、测试
  • 合并条件：CI 测试全绿 + Code Review 同意
  • 合并后自动构建 Docker 镜像并部署环境，随时可回滚

敏捷小组的一周日常

7人典型小队 · 一周一 Sprint · 估算扑克与分支切割

敏捷团队的标准配置与分工

维护某核心业务微服务的一个典型敏捷小组配置：约 7 人
产品经理 (Product Owner)
写需求文档，转化成 Ticket 送入看板背板。
开发人员 (约 4 人，含架构师)
每天从看板 To Do 栏按高优先级领 Ticket 执行。
测试人员 (1 人)
测试环境里跑代码，找 Bug 并报 Ticket。
项目经理 (Scrum Master)
服务型角色，保障流程顾全大局，每周轮值制度。

小型敏捷团队的一周核心节奏（上）

  • 上半周：稳妥部署生产环境
    • 没有程序员愿意星期五发版并在周末心惊胆战修理 Bug。
    • 即便上周代码已测好，生产发布均置于周一/二，预留充足时间试错与回滚。
  • 每周二清晨：迭代回顾会议（Sprint Retrospective）
    • 常规站会结束后开一小时。核心议题：复盘上个 Sprint 哪里好、哪里差。
    • 实战案例：测试抱怨开发发版前乱更代码导致无效测试。团队当场表决决议：非紧急情况严禁在预部署分支更新！

每周四：迭代规划会与“估算扑克”

决定下个 Sprint 做什么以及工作量衡量
  • 从 Backlog 梳理高优 Ticket 开始规划。
  • 评估流程：所有人阅读同一条 Ticket并在脑海评估（1分=1天完工，5分=极难）。
  • 大家一起倒数 3、2、1，同时亮出手势或扑克牌定分数。
估算扑克的好处：
1. 避免项目经理独裁指派，全员积极了解需求内涵。
2. 由实操一线人员给出的分数更加精准服众。
3. 亮牌后，最高分与最低分须当众陈述原因！**强制带动新手小白疯狂学习架构师的排雷经验**。

每周五：分支切割（Branch Cut）

为了下周的稳定测试，隔离主干
周五下午截断：主干继续走下周的新需求，抠出来的测试分支仅留给QA暴力找Bug和验收

怎么保证一周跑一个Sprint，还能保质交付？

背后的三道防线：

  • 极高比例的自动化测试覆盖率。
  • 测试时间空间错位：周五切掉分支，下周开发们开始做新的 Sprint，而测试有一整周时间在此稳定分支环境里慢慢打磨验收。
  • 紧急时刻全员测试，且随时查岗周变更短板。

如何平衡质量与
时间、成本、范围？

软件项目管理金三角 · 瀑布与敏捷的根本分歧

软件项目管理金三角法则

  • 三条边：时间、成本、范围（功能量级）
  • 中间居首：软件工程的核心——质量（永不妥协）
  • 硬核法则：当一条边被外力压缩时（例如时间被老板缩短），另外一条或两条边必须做出改变补偿（如加外包增加成本，或砍掉非核心范围功能）。
  • 如果不做调节改变，内部塌陷的必然是质量之核。

实战博弈：应对职场扯皮场景

老板：“1个月必须上线！”
  • 传统挨打：硬扛说行，疯狂加班熬夜爆肝出废品。
  • 质量法则：接受时间受限。抛出选择：“只留核心功能范围先做个MVP，或帮我申请重金扩两倍人手！”——把难题变选择给老板勾。
产品：“中途强塞一个急客需求！”
  • 由于功能范围（工作体量）发生了剧烈膨胀，不调整必定系统崩溃。
  • 质量法则：坦然接受，交出双选项：“允许原定发版大面积延期，或者向公司讨要经费增加开发人手处理并发。”

为什么瀑布常延期，而敏捷不？（金三角视角）

瀑布锁死了范围，出事只能无边际加班烧钱（时间/成本被动顶飞）。
敏捷锁死了 2 周时间和小组人力（时间/成本铁定），出事就果断砍向低优范围保核心。

经典商业案例：金三角的极致运用

淘宝网 1 个月上线传奇
2003年马云死下命令1个月上线不可减量（锁全须边界）。
开发组绝杀招数：加巨量隐形成本。重金在外面收下成熟交易系统源码再由顶尖黑客改写骨肉。成功狂揽“非典”神级红利。
MVP（最小可行产品）怎么来的？
又想光速快，又想穷省钱没预算（死锁时间/成本边界）。
绝杀招数：无情斩杀功能范围。只做核心业务基点切面放出测试验证。

可行性分析防坑指南

经济 · 技术 · 社会三大维度的无情拷问

为什么常常有人忽略可行性分析？

  • 借口 1：“软件需求起步总是太抽象，无非就是写代码，怎么算？” 真相：可等需求初明后补交分析，苗头不对及时砍断止损。
  • 借口 2：“老板直接拍板强压，我等开发只能认死理执行。” 真相：作为基层需呈交专业冰冷评判求出脱免责；或者老板确实是“马云看阿里云”般的远见布局（你看作不可行，人家看的是十年周期）。
  • 借口 3：“这年头讲究互联网创新试错，搞老一套文档繁文缛节会杀死点子！” 真相：完全可笑。可行性分析是前期拦死无脑作死创新（如使用惹火自焚的违法代码库）的核心护甲。
