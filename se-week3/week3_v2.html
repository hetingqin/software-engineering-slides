第一讲

敏捷开发到底是想解决什么问题？

1. 敏捷开发：与其说是框架，不如说是价值观

很多人对敏捷开发存在误解，认为敏捷就是：
• Scrum 框架或极限编程 (XP)
• 每天雷打不动地开站会、两周一固定的冲刺 (Sprint)
• 用看板软件 (Jira) 管理任务，或向白板上贴便签

事实上：敏捷不是一种工具或框架，也不是一套具体的过程，而是一套价值观和原则。

各种框架和工具是“术”（手段），敏捷本身是“道”（指导决策的思想）。它旨在解决瀑布模型中周期长、发布烦、变更难的三大核心痛点。

2. 敏捷软件开发宣言与底层逻辑 (1/2)

1. 个体和互动 高于 流程和工具
2. 工作的软件 高于 详尽的文档
3. 客户合作 高于 合同谈判
4. 响应变化 高于 遵循计划

注：宣言并非全盘否定右侧价值，但更加看重左侧赋予的灵活性。

2. 敏捷软件开发宣言与底层逻辑 (2/2)

3. 用敏捷的方式“盖房子”（演进式交付）

每次只做最小化可行产品（MVP），然后快速迭代
需求
不再是一本厚厚的需求规格书，而是“作为上班族，我想要一个休息空间”。
前期迭代 (Sprint 1-2)
搭了个草棚（最简可用，客户体验给反馈），然后升级简易厨房，修复屋顶漏水 Bug。
中期迭代 (Sprint 3-4)
升级木屋（自动化测试未完备忘了加窗户），修好窗户，完成砖瓦房。
拥抱变化 (最终)
客户发现空间不够，决定扩建为 3 居室。不再像瀑布模型那样等到大功告成（也可能滥尾）才揭晓。

4. 实施敏捷开发的避坑指南

如果盲目上各种敏捷考核指标却忽略客观规律，敏捷就会沦为“微观管理的镣铐”。实施敏捷开发需满足：

  • 团队规模受控：人数超过两张披萨能喂饱的规模（Two-pizza rule）就要分拆，通常 7±2 人最佳。
  • 紧密沟通与赋能：内部协作透明，领导要支持“自发组织团队”，给工程师自主权。
  • 强健的工程基础：写代码要有自动化测试兜底，否则重构就是“引爆地雷”。
  • 【AI 时代新解】：借助 AI Agent 分配任务、甚至自动根据需求生成测试骨架，缓解“没空写文档和测试”的焦虑。
核心真谛：软件开发最核心的是人。如果不具备全面敏捷的条件（如甲乙双方是死板的外包合同制），可以先局部借鉴持续集成、每日站会等优秀实践。

第二讲

大厂敏捷方法剖析（上篇：核心流程与基建）

1. 分而治之的大型工程策略

大项目拆成小项目

大服务拆成微服务

大团队拆成小团队

在庞大的科技巨头阵营里，项目开发的核心思想是典型的“分而治之”。

底层依赖云原生环境、容器（Docker/K8s）等基础设施提供物理隔离，团队配合依赖标准 API 协议进行协同。

只要组织扁平，庞然大物也能像初创公司一样灵活穿梭。

2. 一切工作围绕 Ticket（工单）开展 (1/2)

  • 无论是报 Bug、提需求，还是重构一段冗余代码——只要是投入时间精力，都必须发起 Ticket。
  • 看板严格划分为：To Do（待办）、In Progress（进行中）、Done（已完成）。
  • 每天上班第一件事，就是对着看板直观掌控 Sprint 的进度。
🚀 AI 赋能实践：现代 Jira 已接入大模型，不仅能对 Ticket 去重，还能自动把一句话需求扩写为包含“鉴权、响应、报错”的标准边界验收条件（AC）。

3. 基于 Git 和 CI 的自动化防线（基石）

大厂敢于一天发布上百次的流水线纪律
🛡️ 代码审查 (Code Review)
不直接提交流到主干 (master)，而是开分支 (branch)。完成后提交 PR，要求同行检视并给出 Approve 才能放行。
🤖 持续集成 (CI)
不知疲倦的机器人。PR 提交后自动校验代码格式、执行安全漏洞扫描、运行全量的单元测试。
⚡ AI Code Review 加入
在 CI 中嵌入 AI 节点，10 秒内给出代码逻辑缺陷、性能隐患的初步评审，减少人类审阅负担。

PR 放行的黄金公式 = CI 绿灯通过 + 人类(或AI) Review 认可

4. 每日站立会议（Daily Standup）的核心效用

站会不是漫长的批斗会和表功会！

必须控制在 15 分钟以内，讲究高效沟通：

  • 灵魂三问：昨天干了什么？今日有什么计划？遇到了什么阻碍无法推进？
  • 动态甄别：梳理新提单的紧急 Bug 和高优需求，安排插队或是放入积压池。
  • 问题停车场 (Parking Lot)：遇强力争论立刻“掐断”停回停车场，会后只拉相关当事人单挑解决。

第三讲

大厂敏捷方法剖析（下篇：一个小组的一周实战）

1. 敏捷团队的标准配置（斯巴达小分队）

维护某个大厂核心业务模块的一个典型 7 人小组
👔 产品经理 (Product Owner)
从业务口径挖掘真实诉求，转化为 Kanban 上的 Ticket。确保做的事情有价值。
💻 开发人员 (约 4 人，含架构)
每天按照优先级从 To Do 栏认领任务，实施代码交付与 Code Review。
🛡️ 测试人员 (1 人)
极高比例地维护自动化测试用例，在测试环境运行代码，找 Bug 报单。
🎯 项目经理 (Scrum Master)
保障流程运转的润滑剂（服务型！而非鞭打型），替开发扫清外围障碍。

2. 精密咬合的“一周开发核心节奏” (1/2)

整个生产链路如同生物钟一般运行
  • 上半周：部署生产环境。永远不要在周五发版！上半周发版能确保故障时全员在工序，随时抢修与回滚。
  • 周二清晨：迭代回顾会议 (Retrospective)。灵魂发问：“哪里好？哪里要抛弃？” 定下的教训红线将被强制执行。
  • 周四清晨：迭代规划会 (Planning)与“估算扑克”。所有人对 Ticket 全盲打分同时亮牌。大差异者辩论，避免专家忽略新手环境耗时，促进知识下沉。
  • 周五下午：分支切割 (Branch Cut)。下班前，从主干切割出一条“预发布分支”交 QA 兜底闭关。主干可继续放飞做新需求。

2. 精密咬合的“一周开发核心节奏” (2/2)

分支切割 (Branch Cut) 演示

3. 三道防线：高压节奏为何不会让产品崩塌？

① 极高比例自动化覆盖
通过率不达标（如 80% 覆盖）被 CI 管道无情拦截，人情和面子统统无效。
② 极短的测试流转空间
由于进行了 Branch Cut，其实测试组有整整一周时间在刚切割的独立测试环境慢慢验收，不受新流进代码的污染。
③ 极细粒度的快速追查
改动小而频，合并极快。一旦出 Bug 可以用 Git 历史光速精确定位引入者。透明度促使全员自检。

第四讲

项目管理的核心博弈——质量金三角

1. 软件工程的“不可能三角” (1/2)

多、快、好、省，你通常无法全部兼得

现实生活和工程领域一样残酷，面对极端的执念，项目管理遵循严酷的金三角法则：

  • 三个顶点/约束边：时间 (进度)、成本 (资源/预算)、范围 (功能规模)
  • 中心与底线：质量（这是系统存活的尊严，轻易不可触碰）
硬核法则：如果客户压缩时间，你要么增加投资开发成本，要么只能砍掉大半产品范围。如果三边锁死还不妥协，最终崩塌的绝对是系统质量。

1. 软件工程的“不可能三角” (2/2)

项目管理金三角图示

2. 面对不合理要求的推诿与破局技巧

场景 A：老板压缩工期（要求 3 个月变 1 个月）

初级应对：硬怼老板，然后无奈疯狂加班，最终造出不可维护的屎山代码崩塌。

金三角解法：发起选择反制！“同意 1 个月，但请允许锁定极度缩小的 MVP 范围（砍功能），或者批下采购成熟组件/拉取巨额外包的预算（加成本）。”

场景 B：产品中途强行塞入重大家需求

现状分析：系统要求开发的范围（体量）突发性变大扩张。

应对方式：微笑接受，但严厉指出金三角形变：“新功能没问题，但旧有的交付日必须**硬性延期**（让时间延展），要么立刻**加人手兵力**（加成本）。”

3. 瀑布 vs 敏捷在金三角博弈上的底层分歧 (1/2)

传统的瀑布路线：固定范围

为执行乙方合同，范围神圣不可侵犯。遇难关为保范围全做完，通常**时间和成本双双失控爆炸**，演变为无尽的拖期死扛。

敏捷路线：固定时间与成本

认为 1 周精力和时间恒定。要兼顾底层质量，只有**将“范围”变为动态裁剪的蓄水池**。排不进的低优先级 Ticket 直接砍飞，这保证了产品版本永远不会得心脏病而跳票。

第五讲

不重视“可行性分析”的悲惨收场

常见放弃可行性分析的不靠谱托辞

为什么它成了大家最懒得写的一份报告？
托辞 1
“需求太模糊了，没办法做前置研究。”

事实：如果连做个预估预演都因为模糊做不到，直接冲进去狂码代码更注定是一场无底洞的盲人摸象！

托辞 2
“老板拍板死任务必须干，分析出不可行有毛用？”

事实：它是帮助你在系统暴雷血案后用来**防范反噬与背锅的免死法宝**，也是促使领导者及时止损的专业理智依据。

托辞 3
“我们在搞互联网敏捷试错，繁文缛节会拖慢进度。”

事实：对敏捷的极大误用。MVP“试错”恰恰是在用极小代价代替全面实战来提前“验证可行性”，而不代表梭哈盲测。

惊心动魄：大厂移动端跨端开发翻车惨案 (1/2)

一场未能穿越可行性三维审查的悲剧

背景：某一线互联网战区决定全面将移动端底座重构成 React Native (RN) 体系，以图节约巨额 Android/iOS 原生人员独立开发产生的维护折损开销。

A 面：经济可行性调研（全员兴奋）

10 人发轫，历时半年的重铸期，人力与硬件开销超百万元。但只要两端合一，预计下半年就能省下巨量双端开发人力并回撤盈利。

结论：从经济账面看，天长地久，极其稳妥可行！

B 面：技术可行性调研（游刃有余）

架构师摸底了解了 RN 的组件库生命周期。虽然少数老旧安卓机的渲染出现卡死（水土不服），以及底层视频流插件需硬刚攻坚，但不存在彻底拿不下的理论盲区。

结论：精锐出马，技术没有生死死穴，安全通过！

惊心动魄：大厂移动端跨端开发翻车惨案 (2/2)

💀 惨绝人寰的 C 面：社会（法律）可行性底线被无情击穿

大部队即将封顶之际，法务天团实施全量溯源盘点，发掘出夹在协议栈里的恶霸条款——**Facebook 主推的“BSD + 专利”捆绑协议**。

这意味着：你可用开源，但若大厂任意出海业务惹起与 Facebook 的任何专利摩擦，自家主 APP 立刻丧失全部授权。这是将命脉直接交由美国硅谷捏握。

尾声：百万元化为齑粉的落幕

无论是开发者的架构尊严还是先前的宏大愿景系统均被绞杀粉碎。整个部门的心血代码只能悉数推翻、连根拔起全部销毁。

**若高管在点亮开发库前，申请哪怕 20 分钟让法务排雷（合规可行性核验），耗时 10 个月的血泪史便可止步。**
