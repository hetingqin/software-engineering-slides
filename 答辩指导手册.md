# 《软件工程及体系结构》期末团队项目答辩指导手册

**课程名称**：软件工程及体系结构
**授课对象**：电子信息工程专业硕士
**答辩时间**：第 15 - 16 周
**答辩形式**：现场/线上（以通知为准）

---

## 一、答辩概述与要求

期末团队项目答辩是检验同学们在这 16 周内对软件工程理论、方法、工具以及 AI 辅助技能综合运用能力的核心环节。答辩不仅看重最终产出的代码和软件，更看重**整个软件工程生命周期的严谨性、团队协作的真实性以及应对技术难度和风险的思考深度。**

### 1. 基本要求
*   **全员参与**：团队所有成员必须出席答辩，如有特殊情况需提前请假并出具证明。
*   **时间分配**：**每组总时长严格控制在 25 分钟**。
    *   **项目陈述与演示（20 分钟）**：由团队代表（可 1-2 人）进行 PPT 汇报及作品演示。
    *   **评委提问与答辩 (Q&A)（5 分钟）**：全体成员在线/在场，回答评委提问。建议分工准备不同的问题领域。
*   **材料提交**：答辩前两天需提交以下电子版物料至指定邮箱/平台：
    1.  答辩 PPT 文件 (`.pptx` 或 `.pdf`)
    2.  项目核心文档（需求规格说明书、设计文档、API 文档、测试报告等，可合并为一个压缩包）
    3.  源代码仓库链接（GitHub / Gitee / GitLab），需开放只读权限给评委
    4.  AI 辅助开发体验报告（团队版一份，不少于 1500 字）

---

## 二、答辩模板与汇报要点

汇报 PPT 建议在 20-30 页之间。重点不在于念干瘪的文字，而在于生动、逻辑清晰地展示"你们是怎么从 0 到 1 用软件工程的方法搞定这个项目的"。

### 汇报结构建议（20 分钟）

#### 1. 项目简介与背景 (2-3 分钟)
*   **团队介绍**：列出成员及各自的核心职责分工（要求体现合理的角色划分，如 PM、研发、测试、架构等）。
*   **项目灵感与痛点**：为什么要做这个项目？解决了什么人的什么问题？（体现需求可行性分析）
*   **产品 MVP**：用一两句话或一张图清晰概括产品的核心价值主张（Minimum Viable Product）。

#### 2. 需求分析与产品设计 (3-4 分钟)
*   **需求挖掘**：展示你们是如何获取和梳理需求的（如 KANO 模型分析、用户访谈结论）。
*   **原型与交互**：展示核心功能的低保真或高保真原型交互图，解释为什么这样设计。
*   **应对变更**：如果在开发中遇到了重大需求变更，你们是如何利用流程控制或者妥协方案解决的？

#### 3. 架构设计与技术选型 (3-4 分钟)
*   **整体架构图**：必须有一张清晰的系统架构图（前端、后端、数据库、云服务等），说明请求流转过程。
*   **技术栈选择**：说明使用了什么技术栈，**重点解释“为什么选它”**（不要只列名词，要讲出选型逻辑和对比过程）。
*   **关键技术难点突破**：分享 1-2 个开发过程中遇到的最棘手的技术难题，以及你们是如何通过排障三步法或工程思维解决的。

#### 4. 软件工程实践与协作 (4-5 分钟) 【核心加分项】
*   **敏捷开发与计划安排**：展示你们的甘特图、燃尽图或看板工具截图，说明你们是如何进行跌代拆分的。
*   **代码协作与版本管理**：如何使用 Git 和 GitHub Flow？是否有合理的分支策略（如 master/dev/feature 等）？是否有 Code Review 机制？
*   **质量保障体系**：有没有写单元测试/集成测试？测试覆盖率多少？遇到 Bug 是如何通过 Issue 管理工具追踪至闭环的？如果是全员测，Endgame 测试是怎么跑的？
*   **持续交付（加分）**：是否部署了简单的 CI/CD 流水线（如 GitHub Actions 自动打包部署等）。

#### 5. AI 应用与研发人效 (2-3 分钟) 【特色模块】
*   **AI 提效成果**：在哪个环节（需求分析、写代码、排查 Bug、生成图表）使用了 AI（如 ChatGPT、Claude、Cursor）？
*   **踩坑与避坑指南**：AI 输出了哪些垃圾代码或产生了什么负面影响？你们是如何通过人工审查或构建自动化防线纠正它的？
*   **生产力统计**：估计项目中约有多少比例的代码/文档是 AI 生成的，效率提升了几何？

#### 6. 产品现场演示 Demo (2-3 分钟)
*   **Live Demo**：现场演示产品的 1-2 条核心业务链路操作。若担心现场网络翻车，可提前准备**完整清晰的录屏视频**。

#### 7. 复盘与总结 (1 分钟)
*   **得与失**：如果项目重来一次，你们会改变哪些技术架构或管理决策？
*   **致谢**。

## 三、评分标准 (总计 100 分)

答辩评分将围绕以下六个维度展开，不仅考核技术，更看重工程化程度。

| 维度 | 分值权重 | 考察核心细项点 | 预期表现（高分特征） |
| :--- | :--- | :--- | :--- |
| **功能完成度与产品价值** | **25%** | • 是否实现了 MVP 核心功能<br>• 需求是否真正解决痛点<br>• Demo 演示的流畅度与视觉效果 | 核心主干流程闭环且无阻塞 Bug；UI UX 设计令人眼前一亮；具有现实使用价值而非玩具。|
| **代码质量与技术深度** | **20%** | • 架构设计是否合理、可扩展<br>• 代码组织、高内聚低耦合度<br>• 核心难点突破程度 | 架构图清晰专业；关键算法或业务逻辑复杂且实现优雅；数据库设计符合规范范式；妥善处理并发或安全隐患。|
| **软件工程实践实施** | **25%** | **(核心重中之重)**<br>• Git 版本控制规范度<br>• 是否有看板/Issue追溯<br>• 文档详实度<br>• 是否有测试策略与 CI 部署 | 有完善的分支结构甚至 PR 记录；详实的燃尽图与任务拆解证明进度可控；具备单元测试并能展示流水线自动部署痕迹；没有专职测试但有完善的自测清单。|
| **AI 使用的广度与深度** | **15%** | • 是否将 AI 融入了多个开发生命周期环节<br>• 是否对缺陷代码有判别和驾驭意识 | 深入应用 AI 解决了某个复杂的脚手架搭建问题；有清晰的 Code Review 防止 AI 幻觉渗透；写了一篇深刻的 AI 工具避坑体验报告。 |
| **团队协作** | **10%** | • 任务分配公平合理<br>• 分工是否符合软件工程角色划分<br>• 知识传承与互相 Backup | "两个披萨"规模切分明确。能在提问环节证明大家协同作战而非单打独斗；文档里有明确的接口规约。 |
| **汇报质量与台风** | **5%** | • PPT 制作大方专业<br>• 时间把控精准<br>• Q&A 环节应答流畅逻辑清晰 | 汇报人吐字清晰、充满自信；不超时；面对刁钻问题能快速切换到对应开发者解释原理，不慌不乱。 |

## 四、附录：给答辩团队的几点最后建议（防翻车指南）

1.  **关于 Demo**：请务必、务必、务必准备好**录屏预案**。答辩现场可能会遇到断网、服务器宕机、依赖包冲突等墨菲定律制裁。
2.  **关于演讲**：大家是团队在汇报，不要只由一个人背所有稿子。PPT 请勿放满屏幕文字照着念，把文字留作提词器，多用图表说话（流程图、架构图、甘特图等）。
3.  **关于被怼**：老师如果指出 Bug 或是架构缺陷，**绝不狡辩与掩盖**，直接承认不足，并尝试口头补充你们原本考虑过的修复排期或 Plan B。坦诚的技术复盘远胜过诡辩。
4.  **守时**：卡点结束是一种巨大的修养。提前彩排掐表 3 遍。超时会被打断且严重扣分。
