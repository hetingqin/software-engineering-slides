## 第一讲：Agent Skills 体系

### 1. 什么是 Skills？

- Skills = AI Agent 的"能力插件"
- 一组指令、脚本和资源的集合，扩展 AI Agent 的能力
- 文件结构：
  - `SKILL.md`：主指令文件（YAML 头部 + Markdown 正文）
  - `scripts/`：辅助脚本
  - `examples/`：参考实现
  - `resources/`：额外资源

### 2. Skills 中的软件工程原则

| 原则 | 在 Skills 中的体现 |
|------|-------------------|
| **模块化** | 每个 Skill 封装独立能力 |
| **复用** | Skill 可跨项目复用 |
| **封装** | 内部实现对用户透明 |
| **组合** | 多个 Skills 组合完成复杂任务 |
| **接口设计** | SKILL.md 定义输入输出规范 |

### 3. 关键洞察

- 传统软件工程：**编程语言**是封装接口的手段
- AI 时代：**自然语言**成为新的封装接口
- SKILL.md 本质上就是"自然语言的 API 文档"
- 软件工程原则（高内聚、低耦合、单一职责）同样适用

### 4. 案例解析：Slide-Deck Skill

- 输入：Markdown 内容
- 处理：分析结构 → 选择布局 → 生成图片
- 输出：一组幻灯片图片
- 体现了完整的：输入验证 → 处理逻辑 → 输出格式化

## 第二讲：AI 时代的软件质量保障

### 1. AI 代码的四大风险

| 风险 | 描述 | 应对 |
|------|------|------|
| **质量风险** | AI 生成代码可能有隐微 Bug | 代码审查 + 完善测试 |
| **安全风险** | 可能引入安全漏洞 | 安全扫描 + 代码审计 |
| **技术债务** | AI 倾向粘贴式编码 | 定期重构 + 架构审查 |
| **能力退化** | 过度依赖 AI 导致技能退化 | 保持手写代码能力 |

### 2. AI 代码审查规范

- AI 生成的代码 **必须经过人工审查**
- 审查重点：
  - 逻辑正确性：AI 是否真正理解了需求？
  - 安全性：是否有注入/泄露风险？
  - 可维护性：代码是否可读？
  - 是否引入不必要的依赖？
  - 测试是否覆盖了边界情况？

### 3. CI/CD 中嵌入 AI 代码扫描

- 在 CI 流水线中增加：
  - 安全漏洞扫描（Snyk, CodeQL）
  - 代码质量检查（SonarQube）
  - License 合规检查
  - AI 代码比例统计

## 第三讲：新时代工程师能力模型

### 1. 能力模型重构

**传统能力模型（编码力为核心）：**
- 算法 → 编程语言 → 框架 → 调试 → 性能优化

**新能力模型（系统能力为核心）：**
1. **架构力**：系统设计不会被 AI 取代
2. **需求判断力**：理解真正的问题比解决问题更重要
3. **AI 协作与审查**：成为 AI 的"导师"而非"学生"
4. **工程思维**：项目管理、风险意识、质量意识
5. **Prompt 工程**：精确表达意图的能力

### 2. "最热门的编程语言是英语"？

- Karpathy："The hottest new programming language is English"
- 辩证分析：
  - ✅ 自然语言确实在成为与 AI 交互的主要方式
  - ⚠️ 但底层编程能力仍然是理解和审查 AI 代码的基础
  - 🎯 最佳策略：**两条腿走路** —— 既会 Prompt，也懂代码

### 3. 核心竞争力

- 技术广度 > 单一语言深度
- 解决问题的能力 > 写代码的速度
- 学习能力 > 已有知识量
- 沟通协作 > 单打独斗
- 工程思维 > 技术极客

### 4. 未来展望

- AI 不会取代工程师，但**会用 AI 的工程师会取代不会用的**
- 软件工程的"道"在 AI 时代更加重要
- "万事皆项目"的思维方式，适用于任何时代
