## 第一讲：新技术与软件工程 + 范式转移导论

### 1. 康威定律

- "设计系统的组织，其产生的设计等同于组织之间的沟通结构"
- 翻译：你的团队组织结构，决定了你的系统架构
- 微服务兴起的组织基础：小团队 → 小服务

### 2. 云计算对软件工程的影响

- IaaS → PaaS → SaaS → FaaS（Serverless）
- 基础设施的抽象层级越来越高
- 开发者越来越少关心"怎么部署"，更多关心"做什么功能"

### 3. 范式转移：从手写代码到 AI 赋能

- **旧模式**：专业人 → 写操作系统 / 写工具 / 写类库 → 程序员使用工具和类库写应用
- **新模式**：AI OS + Skills（能力插件）→ 自然语言驱动 → 应用自动生成
- 核心变化：**封装的接口从编程语言变成了自然语言**
- 这不是取代程序员，而是改变程序员的工作方式

### 4. 软件工程的本质为什么不变？

- 不管用什么工具写代码，**需求分析、架构设计、测试、项目管理**这些阶段不会消失
- AI 改变的是"编码"这个阶段的效率和方式
- 工程思维比任何工具都重要

## 第二讲：Vibe Coding

### 1. 什么是 Vibe Coding？

- 2025 年 Andrej Karpathy 提出"Vibe Coding"（氛围编程）概念
- 核心：用自然语言描述意图 → AI 生成代码 → 人类验收和迭代
- "我不真正写代码，我只是描述、运行、复制粘贴、然后它大多数时候都能工作"

### 2. 行业数据

- GitHub Copilot 超过 2000 万用户
- 调查显示 46% 的新代码由 AI 生成
- 使用 AI 编程工具的开发者效率提升 30-55%

### 3. Vibe Coding 的适用场景 vs 局限性

**适用场景：**
- 原型开发和 MVP
- 前端 UI 开发
- 脚本和自动化工具
- 文档生成
- 学习新技术

**局限性 & 风险：**
- 安全关键系统不能完全依赖
- 可能引入安全漏洞
- 代码质量难以保证
- 技术债务可能更快积累
- 开发者能力退化风险

### 课堂辩论

> **正方**：Vibe Coding 是"开发民主化"，让更多人能创造软件
> **反方**：Vibe Coding 是"技术债制造机"，会产生大量低质量代码

## 第三讲：AI 编程工具生态

### 1. AI 编程工具五层演进

| 层级 | 代表工具 | 能力 | 人的角色 |
|------|----------|------|----------|
| L1 代码补全 | GitHub Copilot | 行级/函数级补全 | 主导编码 |
| L2 对话编程 | ChatGPT, Claude | 对话生成代码片段 | 指导 + 审查 |
| L3 AI IDE | Cursor, Windsurf | 上下文感知、多文件编辑 | 协作 + 审查 |
| L4 AI Agent | Claude Code, Aider | 终端级自主编程 | 监督 + 验收 |
| L5 全自主开发 | Devin, OpenHands | 全流程自主 | 定义需求 |

### 2. 工具选型与团队成熟度

- **新手团队**：从 L1（Copilot）开始，逐步升级
- **成熟团队**：可以使用 L3/L4，配合完善的 Review 和 CI
- **关键原则**：AI 工具是助手，不是替代品

### 3. 课堂演示

- 使用 AI Agent 完成一个小型编码任务
- 展示从 Prompt → 代码生成 → Review → 测试 → 迭代的完整流程
