## 第一讲：新技术爆发与康威定律

### 1. 面对新技术的困惑
微服务、云计算、人工智能……当新技术层出不穷时，单纯从“怎么写代码”去理解会陷入细节。
**软件工程的视角**：跳出技术本身，看它如何改变整个项目的组织框架和协作流程。

### 2. 康威定律 (Conway’s Law)
> **“你设计的系统架构，必然是你组织沟通结构的缩影。”** —— Melvin Conway (1967)

*   **单体庞大系统** 对应着 一个巨型且跨部门沟通困难的大型组织。
*   **微服务架构** 对应着 被打散的一个个敏捷特战小分队（Two-Pizza Team）。

**微服务的本质启示**：
当你犹豫“服务该拆多细”时，不妨先看看你的兄弟团队有多少人。组织结构不支持（如：运维和开发死仇），强推微服务必然走向灾难。技术的架构拆分，本质也是对组织管理权力的重新分配。

### 3. 云计算彻底改变了工程分工
过去：买机器、拉宽带、配防火墙、写底层查库代码、做灾备（全干工程师）。
现在：IaaS -> PaaS -> Serverless（无服务器架构）。

**软件工程意义**：标准化、模块式的“搭积木”成为现实。让开发团队得以将 99% 的精力全部聚焦在**核心业务逻辑**上，底层和脏活累活全部外包给云厂商。

## 第二讲：范式转移与 Vibe Coding 的崛起

### 1. 软件工程的旧模型（专业层层堆叠）
传统的开发价值链：
`系统层（OS） → 工具/框架层 → 类库层 → 应用层`
每一层都需要极度硬核的人才，**代码必须是由受过专业工程训练的人类一行行写出来的**。

### 2. 新模型：大模型成为 "AI OS"
```text
自然语言需求 
   ↓
AI Agent (如 Claude Code / Cursor)
   ↓
调用底层传统框架类库 (FFmpeg/React)
   ↓
自动生成并运行代码
```
**编程接口已经从“特定语法”变成了人类的“自然语言”！**

### 3. Vibe Coding（氛围编程）
由前特斯拉 AI 总监 Andrej Karpathy 提出。
> **工作流**：看问题 → 用自然语言描述提词（Prompt） → AI 瞬间生成代码 → 运行报错 → 把报错信息扔给 AI 自动改 → 跑通。
（人在过程中甚至不需要去逐行看懂那几百行代码的具体逻辑，只沉浸在验证和推演的“氛围”中）。

**行业震撼数据 (2025-2026)**：
*   **46%** 的新生代商业代码由 AI 生成（Java领域高达 61%）
*   **速度提升**：常规任务效率直接飙升 55%

**局限性**：适合快速原型极速验证（MVP）；在金融/航天等安全性0容忍领域依然必须有人类工程师严格兜底。

## 第三讲：AI 编程工具生态全景指南

根据 AI 工具的“自主权”，目前生态呈现清晰的三阶梯梯队：

### 1. L1 级：结对编程助手（AI 补全模式）
*   **代表工具**：**GitHub Copilot**, TabNine
*   **定位**：超级增强版的智能提示。你敲注释，它预测下方代码块。
*   **掌控者**：100% 依然是人类，AI 仅在当前**个别文件**的鼠标光标处提供辅佐。

### 2. L2 级：强上下文 Agent（IDE 原生派）
*   **代表工具**：**Cursor (Composer 模式)**, Windsurf (Cascade)
*   **定位**：不仅能写单行代码，能**理解你本地的几万行跨文件工程全局**。
*   **掌控者**：你可以通过对话，让它一次性穿越并修改 5 个以上的文件来实现一个新功能。

### 3. L3 级：全自主 AI 工程师（CLI/后台流水线派）
*   **代表工具**：**Claude Code**, Devin
*   **定位**：没有炫酷界面的命令行恶魔。你给它一个超复杂的指令，它会自动规划步骤、自动调用终端创建目录、自动装库、自动读文档、反复翻车再自动修 Bug，直到完成。

### 工具选型的工程建议
工具是服务于业务的。不要盲目追求 L3。
*   **小团队/独立开发者原型**：大胆使用 Cursor / Vibe Coding 狂飙。
*   **大厂核心生产环境**：稳妥起步 GitHub Copilot，以**代码审查（Code Review）**为核心防线。
